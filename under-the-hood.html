Under The Hood

This section is intended for engineers who need to debug, maintain, or extend the extension. It covers the architecture, data flow, and implementation details behind each feature.

Tech Stack

  • React + Redux Toolkit — UI rendering and state management
  • Snowpack — ESM-first build tooling (dependencies bundled in snowpack/pkg/)
  • react-json-view — JSON rendering in the result detail panel
  • Chrome Manifest V3 — Service worker, content scripts (isolated + main world), side panel API
  • CSS Custom Highlight API — Search-within-result highlighting
  • File System Access API — Export via native Save As dialog

Architecture Overview

The extension uses Chrome's Manifest V3 architecture with four main components:

  • early-wrap.js — Injected into the page's main world via manifest (world: "MAIN", run_at: "document_start"). Wraps window.fetch and XMLHttpRequest.prototype.open/send before any page JavaScript executes. Sends two-phase messages via window.postMessage.
  • page-script.js — Loaded asynchronously into the main world via a <script src> tag injected by content.js. Handles debug info collection (platform info, nerdpack metadata, nerdlet navigation). Skips fetch/XHR wrapping if early-wrap.js already did it (checks window.__NR1_UTILS_EARLY_WRAP__).
  • content.js — Runs in the isolated world at document_start. Bridges between the main world (page scripts) and the service worker via window.addEventListener('message') and chrome.runtime.sendMessage. Detects SPA navigation by wrapping history.pushState/replaceState. Also executes the "Locate on Page" widget highlight logic directly in the DOM (8 search strategies + scroll-to-lazy-load).
  • background.js — Service worker. Routes messages between content scripts and the side panel via a Chrome runtime port (nr1-utils-panel). Buffers requests, caches debug info, and manages nerdpack metadata fetching via chrome.scripting.executeScript.

Message flow: early-wrap.js (main world) → window.postMessage → content.js (isolated world) → chrome.runtime.sendMessage → background.js (service worker) → port.postMessage → side panel (App.js)

Request Capture & Display

How NerdGraph and NRQL requests are intercepted, parsed, displayed, and exported.

Request Interception (Two-Phase)

The primary interception path uses two-phase messaging via early-wrap.js:

Phase 1 — Request Start

  • Message type: NR1_UTILS_REQUEST_START → forwarded as REQUEST_START by content.js → NEW_REQUEST_START by background.js
  • Sent immediately when fetch() or XHR.send() is called
  • Contains: requestId (unique per request, format: nr1-{counter}-{timestamp}), url, requestBody, startTime (absolute Date.now())
  • App.js processRequestStart() parses the request body to extract the query, variables, and name, then creates a pending entry in Redux with status: 'pending'
  • The UI immediately shows the request with a yellow pulsing timer displaying elapsed time (Date.now() - startTime)

Phase 2 — Request Complete

  • Message type: NR1_UTILS_REQUEST_COMPLETE → forwarded as REQUEST_COMPLETE → NEW_REQUEST_COMPLETE
  • Sent when the response arrives (fetch .then()) or fails (fetch .catch())
  • Contains: requestId, url, requestBody, responseBody, timing (startTime, totalTime, blockedTime)
  • Response body is read via response.clone().text() with a 5-second timeout race — if the body can't be read (cross-origin, very large), falls back to empty string
  • App.js processRequestComplete() calls buildGraphqlRequests or buildNrqlRequests to parse the response, then the Redux completeRequest reducer finds the pending entry by requestId and merges in the response data
  • Status is set to 'success', 'error', or 'timeout' (timeout detected by searching error messages for the word "timeout")

Legacy Single-Phase Path

  • Message type: NR1_UTILS_REQUEST → forwarded as REQUEST_CAPTURED → NEW_REQUEST
  • Used by page-script.js as a fallback if early-wrap.js didn't intercept the request
  • Sends a single message after the response arrives (no pending state)
  • Processed by App.js processRequest() which creates a completed entry directly

Request Parsing

GraphQL

Handled by buildGraphqlRequests.js

  • URL matching: url.pathname.match('graphql')
  • Request body is parsed as JSON. Supports both single objects and arrays (NR1 has used both formats).
  • Response may be JSON or a text stream (detected by data.match(/^id:.*/)). Text streams are parsed by parseTextStream.js.
  • Query name is extracted via regex: /(query|mutation|subscription)\s+([\w]+)/. Fallback: first 24 characters of the query.
  • The full response object is stored as response (not just .data), preserving all fields including extensions, errors, etc.
  • If request body is empty, creates an "Unknown GraphQL Request" entry.

NRQL

Handled by buildNrqlRequests.js

  • URL matching: url.pathname.match('nrql')
  • NRQL queries embedded in GraphQL requests are extracted by extractNrqlFromGraphql.js, which searches variables for strings matching /\bSELECT\b.*\bFROM\b/i patterns.
  • Table name extracted via: /from\s+(\S+)/i

Live Timer

  • Implemented in LogEntry.js using React.useState + setInterval(100ms)
  • Elapsed time calculated as: Date.now() - request.timing.startTime
  • Interval is cleaned up when the request status changes from 'pending'
  • CSS pulse animation on the timing label: @keyframes pendingPulse (opacity 1 → 0.5 → 1 over 1.5s)

Result Sorting

Sorting is applied in both Log.js and App.js (for export consistency):

  • Primary sort: status priority — pending (0) → error/timeout (1) → success (2)
  • Secondary sort: most recent start time first (b.timing.startTime - a.timing.startTime)
  • Error detection: request.status === 'error' || request.status === 'timeout' || !!request.errors

Search

Search for Results (Left Panel)

  • Filters the request list by JSON.stringify(request).toLowerCase().includes(filter)
  • Searches across all fields: query, variables, response, errors, name, timing, etc.

Search Within Result (Right Panel)

  • Uses the CSS Custom Highlight API (CSS.highlights) with Highlight objects
  • findMatchRanges() walks all text nodes in the result container using TreeWalker
  • Two highlight registers: search-results (all matches, yellow) and search-current (active match, orange)
  • Scrolling to matches uses manual container scroll calculation to respect the sticky toolbar height

Export

  • Uses the File System Access API: window.showSaveFilePicker()
  • Exports only selected results from the visible (filtered + sorted) list — not the raw unfiltered data
  • Output is JSON.stringify(selectedData, null, 2)
  • Default filename: {type}-requests-{ISO timestamp}.json

Owning Team Detection

  • Implemented in RequestsPage.js via findOwningTeam()
  • Recursively searches the entire request object for a key named owningTeam
  • Common location in NerdGraph responses: response.errors[].extensions.nrOnly.owningTeam
  • Displayed as a blue info banner above the account ID and error banners

Account ID Identifier Detection

  • Implemented in both RequestsPage.js and Log.js via findAccountIds()
  • Recursively searches the entire request object for keys named account_ids (array type)
  • Common location in NRQL responses: response[].account_ids
  • Detail view: displayed as a green info banner between owning team and error banners
  • List view: when any result has multiple account IDs, a yellow notification banner appears with a toggle to filter to only multi-account results

Request Status & State Management

The extension uses Redux (via Redux Toolkit) for state management. Each request object has a status field:

  • 'pending' — Created by processRequestStart(), waiting for response
  • 'success' — Set by processRequestComplete() when no errors in response
  • 'error' — Set when response contains errors
  • 'timeout' — Set when error messages contain the word "timeout" (case-insensitive)

Key Redux actions:

  • addPendingGqlRequest / addPendingNrqlRequest — Add pending entries to state
  • completeRequest — Finds a pending entry by requestId and merges response data
  • updateGqlRequests / updateNrqlRequests — Add completed entries (legacy path)
  • toggleSelectedIndex / selectAllVisible / clearSelectedIndices — Manage checkbox selection
  • clearLog — Clears all requests for the current tab

Debug Mode

How platform and nerdpack information is collected and displayed.

Platform Info

  • Read from window.__nr by readPlatformInfo() in page-script.js
  • Extracts: version, platformVersion, env, region (from hostname), userId, accountId
  • Multiple fallback paths for userId/accountId: __nr.initialState, URL path (/accounts/\d+), query params, UNSTABLE_app_shell, __hurlState, window.newrelic
  • Re-reads on SPA navigation (1-second URL change poll in page-script.js)

Nerdpack Metadata

  • Fetched via NerdGraph query: actor.nerdpacks.effectiveSubscribedVersions
  • Two fetch mechanisms: page-script.js uses originalFetch with retries; background.js uses chrome.scripting.executeScript with a synchronous XHR as fallback
  • Data refreshed on tabs.onUpdated with status: 'complete' (3-second delay)

Nerdlet Navigation

  • Subscribes to window.UNSTABLE_app_shell.navigationStateContext via .subscribe()
  • Polls for UNSTABLE_app_shell availability (500ms intervals, up to 30 seconds)
  • Tracks mainNerdletId and mainEntityGuid

Cache & Refresh

  • Debug info is cached in background.js (debugInfoCache object)
  • Cache is cleared on page navigation (tabs.onUpdated with changeInfo.url)
  • Panel requests fresh data via GET_DEBUG_INFO message on connection

Widget Correlation & Locate on Page

How NRQL results are matched to dashboard widgets and how the Locate on Page feature works.

Dashboard Widget Map

  • extractWidgetMapFromDashboard() in App.js detects GetDashboardEntityQuery responses by checking if the GraphQL operation name contains "DashboardEntity" or "GetDashboard"
  • Walks response.data.actor.entity.pages[].widgets[] to build an array of widget descriptors: { widgetId, title, pageName, nrqlQueries, layout, inaccessible, dashboardAccountId, dashboardOwnerEmail }
  • NRQL queries are extracted from widget.rawConfiguration.nrqlQueries[].query
  • Widgets with visualization.id === 'viz.inaccessible' are flagged as inaccessible
  • The widget map is stored in Redux state via setWidgetMap action

Widget Matching

  • matchWidgetByNrql(request, widgetMap) in RequestsPage.js matches a captured NRQL query to a dashboard widget
  • Pass 1 — Exact match: Normalizes whitespace and compares lowercased query text
  • Pass 2 — Substring match: Checks if one query contains the other, handling NR1 runtime modifications to query aliases (e.g., 'Slowest 10%' becoming 'Slowest 10% (s) (90%)')
  • Widget titles are also included in the search filter logic, so users can search by widget name

Widget Hint Extraction

  • extractWidgetHints() in extractNrqlFromGraphql.js scans GraphQL variables for known widget-identifying keys: entityGuid, widgetId, dashboardGuid, layoutId, title
  • Also scans for any variable key containing "widget" or "layout" (case-insensitive)
  • Extracts the GraphQL operation name from the query text
  • Attached as widgetHints on NRQL request objects produced by extractNrqlFromGraphql

Locate on Page

Message flow: side panel → background.js → content.js (highlight runs here directly)

  • Panel sends HIGHLIGHT_WIDGET with widgetTitle and widgetId via the port
  • background.js forwards to the active tab's content script via chrome.tabs.sendMessage. If the active tab query fails, it falls back to trying all *.newrelic.com tabs.
  • content.js runs highlightWidgetOnPage() directly (the highlight logic was moved from page-script.js to content.js for reliability — the page script message relay was unreliable)

DOM Search Strategies (tried in order)

  1. Exact leaf-node textContent — Matches elements with no children whose textContent exactly equals the widget title
  2. Exact innerText on small elements — Matches elements under 100px tall whose innerText equals the title
  3. textContent with children (size-bounded) — Matches elements under 60px tall / 600px wide, allowing child nodes (handles NR1 titles with tooltip icons)
  4. Title attribute — Matches elements whose HTML title attribute equals the widget title (handles CSS-truncated titles)
  5. aria-label — Matches elements whose aria-label equals the widget title
  6. startsWith innerText — Matches elements whose innerText starts with the widget title (handles appended icon text like info badges)
  7. React fiber tree — Searches React fiber props (__reactFiber$) on grid/widget/card elements for title, name, or widgetTitle props
  8. TreeWalker text node — Walks all text nodes in the document body for an exact match

Scroll-to-Lazy-Load

If no strategy finds the widget, the extension assumes it is below the fold (NR1 lazy-renders widgets via IntersectionObserver):

  • Finds the dashboard's scrollable container by checking for overflow-y: auto|scroll on grid/dashboard elements
  • Scrolls down incrementally (step size: max(500px, viewport height))
  • Dispatches a scroll event after each step to trigger IntersectionObservers
  • Retries the DOM search after each scroll step (300ms delay between retries)
  • Up to 10 retry attempts before giving up and restoring the original scroll position

Overlay Display

  • Walks up the DOM tree to find the widget container (at least 200px wide and 100px tall, under 90% viewport width)
  • Scrolls the container into view with scrollIntoView({ behavior: 'smooth', block: 'center' })
  • Polls position until scroll settles (3 consecutive stable position checks at 50ms intervals), then fades in the overlay
  • Safety timeout of 2 seconds forces display if scroll never settles
  • Purple 3px border with glow shadow, fades out after 2 seconds

Debug Logging

All Locate on Page operations log to the page console with prefix [NR1 Utils Locate]. When a widget can't be found, closest DOM matches are logged for diagnosis.

Widget Placeholders

  • Generated in RequestsPage.js on each render by comparing widget map NRQL queries against captured requests
  • For each widget NRQL query not yet in logData, a placeholder entry is created with status: 'defined' and _isPlaceholder: true
  • Matching uses normalized comparison: whitespace collapsed to single space, lowercased, then checked for exact equality or substring containment in either direction
  • Placeholders carry _matchedWidget (with title, widgetId, pageName) so "Locate on Page" and the widget banner work on grey entries
  • Displayed at the top of the results list (prepended before live requests)
  • Automatically disappear when a matching live NRQL request is captured
  • Cleared along with all requests when "Clear Log" is clicked

Widget Map Management

  • The setWidgetMap Redux action merges new widgets with existing ones, deduplicating by widgetId. This supports multi-tab dashboards where each tab's widgets arrive in separate GraphQL responses.
  • When the widget map is updated, all existing NRQL requests in state are eagerly re-matched against the new map. This ensures widgets detected after their NRQL queries fired still get correlated.
  • The widget map is cleared when "Clear Log" is clicked on the NRQL tab.

Call Stack Capture

  • captureComponentHint() in early-wrap.js captures new Error().stack synchronously at fetch/XHR intercept time
  • Parses stack frames, skipping internal extension files (early-wrap.js, page-script.js, content.js)
  • Searches for component-relevant patterns: Widget, Chart, Billboard, Table, Line, Bar, Pie, Nrql, Dashboard, Visualization, etc.
  • Extracts the function name from the first matching frame as componentHint
  • Keeps the first 5 meaningful frames as stackSummary
  • Both fields are passed through the message pipeline (content.js → background.js → App.js) and stored on request objects
  • Displayed as a "Source Component" fallback banner in RequestsPage.js when widget map matching does not produce a result

Inaccessible Widget Detection

  • Widgets with visualization.id === 'viz.inaccessible' in the GetDashboardEntityQuery response are flagged
  • For these widgets, rawConfiguration is empty ({}) and linkedEntities is null — the platform intentionally hides all widget details
  • The source account is not exposed in any API response — confirmed via HAR analysis. The viz.inaccessible visualization is a static component with a hardcoded message
  • The amber banner shows the count of affected widgets, lists their names, and displays the dashboard owner's email (from entity.owner.email) as the contact for account identification
